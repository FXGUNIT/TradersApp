## Core Feature Set — What Traders Regiment Does

### The 5-Phase Recursive Consensus Engine

At the heart of Traders Regiment is our proprietary **5-Phase Recursive Consensus Engine** — a multi-model AI architecture where 12+ independent models vote on every trading decision. Not one model. Not two. Twelve.

Each model brings a different analytical lens:
- **Technical Analysis models** — candle patterns, support/resistance, volume profile
- **Options Flow models** — IV regime, gamma exposure, options desk data
- **Session Probability models** — historical session behavior, liquidity patterns
- **Regime Detection models** — HMM, fractal, anomalous diffusion detection
- **Magnitude models** — size of the move, not just direction
- **Alpha generation models** — identifying edge beyond the consensus
- **Direction models** — bull/bear/neutral with confidence scores
- **Mamba Sequence models** — long-horizon pattern recognition

These models don't just vote — they argue. They challenge each other. The ensemble converges on the highest-confidence signal through recursive deliberation.

### Watchtower — The Self-Correcting Guardian

Markets change. Regimes shift. What worked last week can lose money this week.

Watchtower is our **self-correcting system** — a continuously running evaluation engine that monitors:
- Model drift and performance degradation
- Regime changes in real-time
- Signal accuracy vs. actual outcomes
- Performance decay across all model families

When Watchtower detects that a model is drifting — losing accuracy, misclassifying regime, or underperforming — it automatically flags it, weights it down, and triggers a recalibration cycle.

**Watchtower doesn't wait for the trader to notice. It notices first.**

### The Deliberative Board Room

Every signal that comes out of the Consensus Engine must pass through our **Deliberative Board Room** before it reaches the trader.

The Board Room has five members — each a specialized AI agent with a specific role:

| Board Member | Role | Can Veto? |
|---|---|---|
| **Tech_IB** | IB/VA geometry analysis | No |
| **Tech_Candle** | Candle pattern recognition | No |
| **RegimeWatcher** | Market regime detection | **Yes — CRISIS, CONTAGION** |
| **OptionsDesk** | IV regime + premium health | **Yes — EXPIRY, LOW_IV** |
| **RiskOfficer** | Hard risk limits | **Yes — DAILY_STOP, MAX_POSITIONS** |

The Board Room applies a formal consensus rule: **3 of 5 must agree to proceed**. Any single veto — from RegimeWatcher, OptionsDesk, or RiskOfficer — converts the signal to NEUTRAL. No trade.

RiskOfficer always has the final word. The board cannot override it.

**This is what institutional risk governance looks like — now available to every retail trader.**

### Self-Training and Self-Improving AI

Traders Regiment is not a static system.

Every trade — every signal generated, every outcome recorded, every regime the market enters — feeds back into the training pipeline. Our AI uses:
- **Continual Learning** with Elastic Weight Consolidation (EWC) — learns new patterns without forgetting old ones
- **Experience Replay** — replays significant historical events to reinforce learning
- **Paper Trading Validation** — every new model or strategy is paper traded for a minimum 1 trading week before consideration for live use
- **Performance Drift Detection** — automatic alerts when model performance degrades below thresholds
- ** Bayesian Optimization** — hyperparameter tuning that adapts to market conditions

The system gets smarter every day. Not because we push updates — because it learns from use.

### Session Probability Engine

Markets behave differently at different times of day. The opening range is volatile. The lunch block is chop. The close has its own personality.

Our **Session Probability Engine** analyzes historical behavior for each session (pre-market, morning, lunch, afternoon, close, NY session, Asian session, European session) and provides:
- Probability distribution of directional bias per session
- Expected range, typical chop vs. trending behavior
- Session-specific confidence adjustments for the consensus signal
- DST-aware session timing across global markets

### Exit Strategy and Risk/Reward Optimization

Consensus tells you direction. Our **RRR Optimizer** and **Exit Strategy Engine** tell you how to manage the trade:
- Optimal stop-loss placement (based on ATR, recent swing, and regime)
- Three target levels with progressively higher probabilities
- Risk/Reward ratio for each target
- Position sizing recommendation based on account risk parameters
- Exit timing recommendations — when to take profit, when to trail stops

### Alpha Engine

Alpha is the edge that comes from combining models in ways that no single model can achieve alone.

Our **Alpha Engine** identifies non-obvious correlations across:
- Multi-timeframe consensus divergence
- Sentiment vs. price action divergence
- Regime transition alpha (the edge that exists precisely when regime changes)
- Options flow signals that precede price moves

Alpha is not just generated — it's explained. Every alpha signal comes with a human-readable breakdown of why the edge exists.

### Market Regime Detection

The same signal means different things in different regimes.

Traders Regiment continuously classifies the market into one of four regimes:
- **TRENDING_UP** — momentum strategies favored
- **TRENDING_DOWN** — shorting strategies favored
- **CHOP** — range-bound, mean reversion favored
- **CRISIS** / **CONTAGION** — risk-off, no new positions

The regime classification drives a **dynamic strategy weight adjustment** — the consensus engine literally changes how it votes depending on what regime the market is in.

## What Makes Traders Regiment the Smartest

### Best Architecture — Purpose-Built for Speed and Accuracy

We didn't just "use AI." We made deliberate architectural choices at every layer:

**Frontend:** React + Vite — fast, reactive, clean. The trader sees the signal the moment it's ready.

**Backend (BFF):** Node.js — orchestrates all services concurrently. When a consensus call is made, all 12 models are queried simultaneously (not sequentially). The bottleneck is the slowest model, not the sum of all models.

**ML Engine:** Python FastAPI — every model runs in its own isolated process. LightGBM for tabular/structured data. PyTorch (Mamba) for sequence models. scikit-learn for regime detection. Each tool chosen for the specific task it's best at.

**Communication:** REST internally. Structured JSON contracts between every layer. Zero ambiguity at service boundaries.

**Caching:** Redis-ready architecture with circuit breakers on every external call. ML consensus cached at 60s TTL. Regime cached at 300s TTL. The system is fast because it doesn't recompute what it already knows.

### Best Language Selection — The Right Tool for Every Task

| Task | Language/Stack | Why |
|---|---|---|
| Real-time signal orchestration | Node.js | Non-blocking async, concurrent service calls |
| ML model inference | Python (scikit-learn, LightGBM, PyTorch) | Mature ML ecosystem, best-in-class libraries |
| Sequence modeling | PyTorch Mamba | State-of-the-art long-range dependency modeling |
| Regime detection | Python (HMM, fractal analysis) | Statistical foundations for probabilistic regimes |
| Frontend UI | React | Reactive, component-based, fast renders |
| Data pipeline | Python (pandas, numpy) | Best for numerical/financial computation |
| Telegram bridge | Node.js | Event-driven, low latency for alerts |
| Circuit breaking / resilience | Node.js | Built-in async error handling |

### Best Security

Security is not an afterthought — it's architecture:

- **No secrets in code** — everything in Infisical, injected at deploy time
- **Parameterized queries everywhere** — SQL injection physically impossible
- **Rate limiting on all public endpoints** — abuse cannot exhaust resources
- **Circuit breakers on all external calls** — cascading failures impossible
- **Timeouts on every external call** — ML Engine: 5s max, News: 3s max
- **Input validation at every layer** — Pydantic (Python), Zod (Node.js)
- **CORS properly configured** — no wildcard in production
- **SSH key-based auth** on all VPS access — no password authentication possible
- **fail2ban** running on VPS — automatic brute-force blocking
- **UFW firewall** active — SSH locked to known IPs, ports 80/443 open to all

### Best Data Infrastructure

- **SQLite candle database** — efficient, portable, ACID-compliant
- **DVC tracked datasets** — every dataset versioned alongside code
- **MLflow experiment tracking** — every training run logged with parameters, metrics, and model artifacts
- **DVC + MinIO** — S3-compatible model store, reproducible from scratch
- **Self-improving data pipeline** — new candles feed feature engineering automatically
